# 中国专题地图制图 Skill

这个 skill 用于把"中国范围专题地图"任务沉淀成一个稳定工作流。它适合的任务包括：

- 高亮若干中国城市或行政区
- 基于本地 shapefile 出图
- 使用本地栅格底图并做裁切、重投影
- 添加南海插图和九段线
- 绘制省界、国界、经纬网、比例尺、指北针、图例
- 调整配色、透明度、边界线样式、插图布局

## 附带脚本

本 skill 的 `scripts/` 目录包含以下可直接使用的脚本：

| 脚本 | 用途 |
|------|------|
| `scripts/download_basemap.py` | 下载 Natural Earth 50m 全球底图（HYP_50M_SR_W） |
| `scripts/download_land_border.py` | 下载 Natural Earth 10m 陆上国界线数据 |
| `scripts/map_cities_with_inset.py` | 完整示例：高亮 8 个城市 + 南海插图 + 九段线 |

当用户需要下载底图或国界线数据时，直接引导他们运行 `scripts/` 下的对应脚本。

当用户需要生成中国专题地图时，参考 `scripts/map_cities_with_inset.py` 的代码组织方式（配置区 → 数据准备 → 像素转换 → 绘图 → main 入口）。

## 何时使用

当用户出现下面这些意图时，应优先使用这个 skill：

- 让 Claude 帮忙生成中国地图或中国专题图
- 想高亮中国城市、省份、边界线、九段线
- 明确提到南海插图、比例尺、经纬网、国界、省界
- 想把已有中国地图脚本继续微调
- 想把地图换成某个投影坐标系后输出
- 想基于本地行政区划数据和本地底图出图
- 说"底图没显示""地图还是白的""图层对不上""地图错位了"
- 只想改一小块，例如：留白、插图位置、比例尺、图例、标注、边界线样式
- 明确说"不要重写整张图，只在原脚本上改"

## 目标输出

默认输出应包含：

- 一份可运行的 Python 绘图脚本
- 一张最终地图 PNG
- 若用户需要，可附带下载脚本、底图说明、参数说明

如果用户已经有现成脚本，默认优先在原脚本上继续修改，而不是无必要地整份重写。

## 输入检查

开始写代码前，先检查下面这些输入是否齐全：

1. 行政区或边界数据

- 是否有省级 / 地级 / 国界 / 九段线 shp
- 是否明确知道文件路径

2. 底图

- 是否已有本地底图文件（如 tif + tfw）
- 如果没有，是否需要先下载公开底图

3. 地图范围

- 主图范围是否明确
- 南海插图范围是否明确

4. 投影要求

- 是否明确要求投影坐标系
- 如果要求投影坐标系，底图也必须同步重投影

5. 输出要求

- 是否只要脚本
- 是否要脚本 + PNG
- 是否要保留参数区和中文注释

如果这些信息有缺口，应先补齐最关键的部分，再继续写图。不要在路径、投影、底图都不清楚的情况下直接开始堆代码。

## 缺失信息处理规则

当输入不完整时，按下面规则处理：

### 必须先问清楚再继续的情况

- 用户明确要求使用某个本地文件，但路径不完整或文件不存在
- 用户明确要求某个投影坐标系，但没有说明主图和底图是否都要跟着投影
- 用户要求输出最终地图，但没有任何行政区划数据或边界数据来源

### 可以先用保守默认值继续的情况

- 用户没有明确主图范围时，可以先用常见中国主图范围做默认值
- 用户没有明确南海插图范围时，可以先用常见南海范围做默认值
- 用户没有明确配色时，可以先用高区分度专题配色

### 一旦使用了默认值，应明确告诉用户

- 你用了什么默认值
- 这些默认值在哪个参数区可以继续改

## 维护模式输入检查

如果用户不是让你从零开始画图，而是让你"在已有脚本上继续改"，先检查下面这些内容：

1. 现有脚本路径

- 用户要改的是哪一个脚本文件
- 不要自己猜错文件名

2. 现有输出文件名

- 用户是否希望继续覆盖原来的 PNG
- 如果用户已经有固定输出文件名，默认保留，不要擅自换名

3. 用户希望"只改哪一块"

- 是只改底图
- 只改边界线样式
- 只改九段线
- 只改布局留白
- 只改图例、比例尺、指北针、标注

4. 用户不希望你动什么

- 是否要求保留原脚本结构
- 是否要求保留原函数名
- 是否要求不要重写整张图

5. 是否已经有当前可运行版本

- 如果已有脚本能正常出图，优先做最小修改
- 如果已有脚本连图都出不来，先修运行链路，再谈样式

在维护模式下，默认优先策略是：

- 先改参数区
- 再改局部函数
- 最后才考虑整体重构

除非原脚本已经明显失控，否则不要一上来重写全部逻辑。

## 推荐工作流

按下面顺序处理，不要跳步：

1. 先确认输入数据

- 明确用户提供的是哪些 shapefile、底图、投影要求
- 如果有本地底图，优先使用本地底图
- 如果没有本地底图，可考虑先下载一个可公开使用的底图，再接入脚本

2. 明确绘图范围和投影

- 区分主图范围与南海插图范围
- 明确最终出图到底是地理坐标系还是投影坐标系
- 如果矢量层投影了，底图也要跟着重投影，不能只裁切不重投影

3. 先保证底图对齐，再叠加专题层

- 底图问题优先于样式问题
- 如果底图和矢量不对齐，先修投影和栅格重采样
- 不要在底图错位的情况下继续堆叠样式细节

4. 分层绘制

通常建议按这个顺序：

- 底图
- 经纬网
- 省界
- 国界
- 九段线 / 国界色带
- 城市面或专题面
- 城市标注
- 图例 / 比例尺 / 指北针
- 南海插图

5. 样式优先保证"可读性"

- 重要边界应比次要边界更显眼
- 城市标注如果压在复杂底图上，应给轻量掩膜
- 图例只保留用户关心的元素
- 如果一个样式很复杂但不清晰，优先简化而不是继续叠加效果

6. 改动时优先最小化

- 如果用户已经有脚本，优先保留原结构
- 尽量把修改限制在相关函数或参数区
- 如果只是微调样式，不要重写整张图

7. 如果用户是在"修已有脚本"，优先保留原结构

- 先定位相关函数或参数区
- 能局部修改就不要重写整份脚本
- 只有原脚本结构已经明显不可维护时，才考虑整体重构

## 代码组织建议

生成脚本时，优先整理成下面这种结构：

- 配置区
- 数据准备函数
- 底图裁切与重投影函数
- 像素坐标转换函数
- 绘图函数
- 主图函数
- 插图函数
- `main()` 入口

这样用户后续自己维护时，只需要改顶部参数区，而不必在脚本里到处找数字。

- 主图范围
- 插图范围

如果脚本超过中等复杂度，建议把参数区写成清晰的小节，例如：

- 路径参数

- 底图扩边参数
- 城市专题参数

## 重要原则

### 1. 先解决空间对齐，再解决美观

如果地图底图和矢量层不对齐，先修：

- 坐标系
- 投影范围
- 底图重采样

不要先花很多时间调颜色、线宽、透明度。

### 底图故障排查清单

如果用户说"底图没显示""底图是白的""底图和矢量对不上"，按这个顺序排查：

1. 底图文件路径是否真实存在
2. 底图是否真的被读入，而不是读空数组
3. TIFF 与世界文件是否匹配
4. 裁切范围是否正确
5. 底图是否真正做了重投影，而不是只改了显示范围
6. 栅格和矢量是否在同一目标坐标系下叠加
7. `imshow` 或像素映射链路是否把内容画到错误位置
8. NoData / 透明度 / 显示拉伸是否导致"看起来像白图"

优先把这条链路跑通，再调样式。

### 2. 对中国地图要特别注意南海插图和九段线

如果用户明确要求南海插图或九段线，就不要省略。南海插图应尽量与主图风格一致，避免像两个完全不同的图拼在一起。

### 3. 代码要便于后续手动微调

很多用户最终都会自己改数字。所以：

- 重要参数要集中在顶部
- 参数要有中文注释
- 不要把控制布局的数字散在一大堆函数里

### 4. 优先让脚本"可维护"

如果已经形成了稳定流程，优先整理成：

- 配置区
- 数据准备函数
- 几何处理函数
- 绘图函数
- `main()`

这样用户以后自己改时，不需要在一个几百行的大函数里找数字。

## 处理边界线和九段线的经验

### 国界线

如果用户想要"国界线外侧有颜色带"，优先考虑：

- 先按边界线生成外侧带状区
- 再叠加黑色核心边界线

如果用户强调"色带只能朝外，不要包住整条线"，不要直接用普通粗描边，应考虑缓冲带或单侧缓冲。

如果用户给了专门的"陆上边界线 shp"，优先沿这条线本身做外扩色带，而不是沿整个中国面外轮廓去做。

### 九段线

九段线通常比普通边界线更难处理，因为它是离散短线。若用户希望九段线也有外扩色带：

- 优先使用单侧缓冲
- 必要时给每段线做端点延长，避免紫色带比黑线短

如果用户要求九段线和国界线风格统一，应优先统一：

- 颜色体系
- 外扩逻辑
- 透明度层次
- 黑色核心线宽度

## 处理标注和图面元素的经验

### 经纬网与经纬标注

- 经纬线要画满到边框附近
- 标注要和实际经纬线位置对齐
- 如果某些标注超出图框，应直接不显示

### 城市标注

- 若底图复杂，应给文字加轻量底框或掩膜
- 掩膜不要太重，避免影响地图整体观感

### 图例

- 图例只保留用户关心的元素
- 如果用户不想看到某条线或某个符号，不要强行保留在图例里

## 失败优先级

当任务中同时存在多个问题时，按下面顺序处理：

1. 路径错误 / 文件不存在
2. 坐标系不一致 / 投影错位
3. 底图没有正确显示
4. 南海插图和九段线缺失
5. 图面样式问题

也就是说，如果地图都没对齐，不要先去调颜色或图例位置。

## 底图获取与缓存

如果用户没有现成底图，按下面顺序处理：

1. 优先使用本地底图

- 如果用户已经给了 tif、tfw、jpg+世界文件、或其他本地底图文件，优先直接使用
- 不要在已有本地底图可用时，擅自换成在线底图

2. 没有本地底图时，再考虑下载公开底图

- 优先选择公开、稳定、容易本地缓存的底图资源
- 如果是专题地图，优先考虑能落地到本地文件系统的栅格底图，而不是只依赖在线瓦片

3. 下载后要本地缓存

- 不要每次出图时都重新下载底图
- 应把压缩包、解压目录、实际使用的 tif 路径写清楚
- 如果后续用户还会继续改图，最好额外给一份下载脚本

4. 代码里要明确底图来源

- 脚本顶部要能看出底图文件路径
- 如果底图是下载来的，最好在代码注释里写清楚来源
- 如果用户后续需要替换底图，应尽量只改顶部路径参数

5. 底图更换策略

- 如果用户只是想让底图显示范围更大，通常优先改裁切范围和扩边参数，不要急着重新下载
- 只有在当前底图分辨率、风格或覆盖范围明显不合适时，才考虑换底图源

6. 底图缓存后的推荐交付

- 最终地图脚本
- 本地底图文件路径说明
- 如果是你下载的底图，最好保留下载脚本
- 如有必要，可补一个简短说明文档，说明底图从哪里来、脚本实际用的是哪个文件

## 默认交付风格

除非用户明确反对，否则默认应：

- 用 Python 脚本出图
- 保留顶部配置区
- 保留中文参数注释
- 生成最终 PNG
- 不保留无用测试输出和调试文件

## 输出要求

如果用户没有另外要求，默认应：

- 生成可直接运行的 Python 脚本
- 生成最终 PNG 地图
- 在代码中保留中文参数注释

如果任务本质上是"修已有脚本"，默认应保留：

- 原脚本文件名
- 原输出文件名
- 原有能复用的函数结构

如果用户要求"方便后续自己改"，应额外保证：

- 关键样式参数在顶部配置区可直接修改
- 关键函数有中文职责注释
- 输出目录里不要留下大量无用调试文件

## 示例任务

**示例 1：**
用户要基于中国行政区划数据，高亮 8 个城市，并加南海插图、九段线、省界、国界、比例尺和指北针。

应输出：

- 一个完整的 Python 制图脚本
- 一张最终成图 PNG
- 顶部配置区中包含范围、颜色、比例尺、插图位置等参数

**示例 2：**
用户已经有一张中国地图脚本，只想把九段线改成带紫色色带的外扩风格。

应优先：

- 保留原有脚本结构
- 只修改九段线相关的几何处理和绘制函数
- 不去无关地重写整张图

**示例 3：**
用户说地图底图和矢量不对齐。

应优先检查：

- 矢量坐标系
- 底图裁切是否基于同一范围
- 底图是否真正做了重投影，而不是只改显示范围

## 不该做的事

- 不要在底图错位时继续调样式细节
- 不要把所有控制参数写死在函数内部
- 不要为了"好看"把图例、比例尺、插图压到主图主体上
- 不要默认删掉南海插图或九段线这类用户明确要求的要素
- 不要在已有脚本上做与用户目标无关的大规模重写
- 不要保留明显无用的测试脚本、调试图和中间文件

## 结束时应交付什么

通常至少交付：

- `map_xxx.py` 这样的脚本
- `xxx.png` 地图成图

如果用户后续还要维护，最好额外交付：

- 参数说明
- 底图下载脚本
- 相关数据路径说明

## 本 Skill 附带脚本说明

本 skill 的 `scripts/` 目录提供了可以直接使用的 Python 脚本。当用户需要以下操作时，引导他们使用对应脚本，而不是从头编写：

- **下载底图**：运行 `python scripts/download_basemap.py`，会从 Natural Earth 下载 HYP_50M_SR_W 全球地势底图并解压到项目目录。用户需要在脚本中修改 `WORKDIR` 变量指向自己的项目目录。
